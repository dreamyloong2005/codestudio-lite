# Findings

## Research Notes

- MEMORY.md points Codex++ plugin unlock to `assets/inject/renderer-inject.js`, separate from provider sync, using CDP injection.
- codestudio-lite launches Codex with `--remote-debugging-port` / `--remote-allow-origins`, then when `patchForcePluginUnlock` is true runs a one-shot CDP `Runtime.evaluate` script.
- codestudio-lite's injected script force-enables the plugin nav entry by spoofing React auth context to `chatgpt`, and force-enables disabled install buttons through DOM/React-prop cleanup plus a 1000 ms refresh loop.
- codestudio-lite does not contain the Codex++ modern marketplace unlock layer: no `list-plugins`, `install-plugin`, `marketplaceKinds`, `app-server-manager-signals`, or `Array.prototype.filter` plugin-marketplace patch in its Codex client code.
- Codex++ uses a richer bridge: `Page.addScriptToEvaluateOnNewDocument` plus immediate `Runtime.evaluate`, a backend binding, and a long-lived message pump.
- Codex++ has three plugin unlock layers: legacy entry unlock, modern marketplace request/filter/result patch, and force install button unlock. It also selects legacy/modern behavior by Codex app version and disables plugin patching in relay mode.
- Implemented codestudio-lite's missing modern marketplace layer behind the existing `patchForcePluginUnlock` setting: module discovery for `app-server-manager-signals-`, `list-plugins` request expansion by deleting `marketplaceKinds`, `install-plugin` marketplace-name restoration, marketplace display-name patching, and targeted `Array.prototype.filter` bypass for Codex marketplace/build-flavor filters.
- Upgraded codestudio-lite CDP injection to register the script with `Page.addScriptToEvaluateOnNewDocument` before the immediate `Runtime.evaluate`, matching Codex++'s persistence pattern more closely.
- Live Windows update evidence on 2026-06-23: `latest/manifest` reports x64 `packageMoniker` `OpenAI.Codex_26.616.9593.0_x64__2p2nqsd0c76g0`, while `latest/checksums` lists the arm64 `.Msix` before the x64 `.Msix`. Current `checksum_for_windows` accepts the first `.Msix`, so it can select the arm64 SHA for an x64 download.
- `https://codexapp.agentsmirror.com/latest/win` redirects to an x64 `.msix` response (`Content-Disposition: filename="Codex-Windows-x64.msix"`, `Content-Length: 667793718`), and the downloaded file SHA-256 is `d0d57caccde4e95b6326c8ab8f5ebb610cbbaff4f80197d34e586432d57ad84d`, matching the x64 checksum entry rather than the first arm64 entry.
- User confirmed on 2026-06-23 that Windows Claude Desktop exposes the same official main-process debugger path as macOS. The Windows localization redesign should therefore use debugger/runtime injection only and should remove the old `app.asar` rewrite, Electron fuse flipping, and elevated PowerShell patch fallback entirely.
- Current `src-tauri/src/core/claude_desktop_patch.rs` still contains the old Windows scheme: `launch_windows_claude_desktop(localize)` calls `apply_localization_patch()` and `activate_localized_claude()`, and `ensure_localization_patch()` delegates to `apply_localization_patch()` on Windows. The same file also contains old helpers for resolving Claude install paths, building patched ASAR content, flipping fuses, and running elevated patch scripts.
- Windows Claude Desktop localization now uses the same non-destructive shape as macOS: prepare local runtime/locale files, enable Claude Desktop Developer Mode, launch the official app by MSIX/exe activation, open the official main-process debugger route, and inject through the Node inspector. The old Windows app.asar/fuse/elevated patch implementation blocks were removed rather than kept as fallback.
- Direct Claude Desktop page launch performs debugger enablement and runtime injection synchronously; external terminal launches still use the background injector because the terminal owns the actual app launch process.
- Windows direct localized launch must call `ensure_patch_files()` before writing `localized-launch.flag` and injecting. Without that preparation, `translation-runtime.js` and the zh-CN locale payloads may not exist, so the Node inspector injection cannot install the language whitelist/radio fixes and Claude appears unable to select Chinese.
- Windows direct localized launch must not synchronously call `ensure_windows_claude_main_process_debugger()` and `retry_inject_localization()` from the Tauri command. That path can wait up to the debugger timeout and makes CodeStudio Lite look frozen; the correct shape is to prepare runtime files, launch official Claude, return, then inject in a background thread.
- Claude Desktop Windows uninstall should not use `winget uninstall --id Anthropic.Claude --exact`. On this machine the observed exit code `-1978335210` is `0x8A150016`, which corresponds to a multiple-package match from winget. The safer path is to dispatch by install kind: MSIX/AppX through direct `Remove-AppxPackage` using detected package identities, native EXE through the registry uninstall script.
- On 2026-06-24 this machine no longer reports a registered Claude Desktop MSIX via `Get-AppxPackage`, but still has a native EXE install registered at `HKCU:\Software\Microsoft\Windows\CurrentVersion\Uninstall\AnthropicClaude`, version `1.14271.0`, under `C:\Users\dreamyloong\AppData\Local\AnthropicClaude`. Therefore a selected MSIX uninstall can legitimately complete while global Claude detection still reports installed because the EXE install remains.
- Claude Desktop Windows uninstall verification must follow the selected/effective install kind. MSIX uninstall success should require only that no registered MSIX remains; stale `WindowsApps` residue should not fail the uninstall. EXE uninstall success should require only that the native EXE install is gone; if registered MSIX remains, or only AppX residue remains, the success message should distinguish that from a failed EXE uninstall.
- The remaining `C:\Program Files\WindowsApps\Claude_*` directories are complete MSIX package payloads, but on this machine they are not registered for the current user. Normal Claude Desktop installed-state detection must not fall back to these `appx-stale` directories or show `Resolved: ... (appx-stale)` as installed. Those stale manifests should remain available only to `repair_claude_msix_registration()` when a launch explicitly chooses the MSIX route.
- Claude Desktop Windows uninstall now treats file deletion as part of success. The MSIX path removes the registered AppX package, then scans `C:\Program Files\WindowsApps` for directories that match Claude package identities, the Anthropic publisher suffix, `AppxManifest.xml`, and `app\Claude.exe`; it attempts normal removal first, then an elevated cleanup with the same safety checks, and reports failure if files remain or WindowsApps cannot be verified. The native EXE path collects registry `InstallLocation` plus common local install roots, removes them only when they are under allowed roots and look like Claude, then exits non-zero if any install root remains.
- The final stubborn residue was not AppX registration itself but Claude Desktop's background service: `CoworkVMService` (display name `Claude`) kept `cowork-svc.exe` alive from the partial MSIX payload. After `Remove-AppxPackage`, the WindowsApps directory could shrink to only `app\resources\cowork-svc.exe`, so the previous cleanup missed it because it required a complete package (`AppxManifest.xml` plus `app\Claude.exe`) before deletion.
- Current machine state after manual cleanup on 2026-06-24: `Get-AppxPackage -Name '*Claude*'` returns no package, `C:\Program Files\WindowsApps` has no `Claude_*` directory, and no `cowork-svc` process is running; `CoworkVMService` is still registered as stopped/Automatic because deleting a service with `sc.exe delete CoworkVMService` requires elevation and the current shell is not administrator.
- Windows Claude Desktop uninstall now stops `CoworkVMService` / `cowork-svc` before removing MSIX or EXE files, recognizes partial safe Claude package directories that contain `app`, `app\resources`, or `app\resources\cowork-svc.exe`, and attempts service deletion with a normal `sc.exe delete` followed by an elevated PowerShell fallback. If the service remains, uninstall is reported as failed instead of pretending the machine is clean.
- On 2026-06-24, the latest official Claude Desktop Windows MSIX endpoint redirects to version `1.15200.0`, while this machine still had a stale native EXE uninstall registry entry at `HKCU:\Software\Microsoft\Windows\CurrentVersion\Uninstall\AnthropicClaude` reporting `1.14271.0` and pointing at a missing `C:\Users\dreamyloong\AppData\Local\AnthropicClaude` install root. The app was downloading latest correctly; stale Windows uninstall state could still surface the old version.
- Claude Desktop Windows install/update should remove stale deprecated EXE uninstall registry entries before installing the official MSIX. The cleanup must keep live EXE installs whose `InstallLocation` still contains `Claude.exe` plus `resources\app.asar` or a valid `app-*` image, and it should not block MSIX install if an elevated HKLM stale key cannot be removed.
- Claude Desktop should no longer participate in winget update detection or return a winget update command on Windows. The detector should use the official MSIX install wording and the official Claude latest-release cache for version/update state.
- Windows Claude Desktop Main Process Debugger automation cannot reuse the macOS top-menu mental model. The current script depends on `MainWindowHandle` and sends `Alt+D`, `{DOWN}{ENTER}`, `{ENTER}`; live testing showed hidden/running Claude processes can all report `MainWindowHandle=0`, and after explicit MSIX activation the window is a `Chrome_WidgetWin_1` surface with an in-window hamburger button rather than a Windows top menu. Keyboard attempts with `Alt`, `F10`, `Alt+F`, and `Alt+D` did not open the Node inspector on port 9233.
- The live official Windows Claude Desktop menu path works through UI Automation: opening the in-window hamburger menu exposes `Developer` and `Enable Main Process Debugger` menu items, and toggling `Enable Main Process Debugger` opens a Node inspector at `http://127.0.0.1:9229/json`. The returned target is `electron/js2c/browser_init` with `webSocketDebuggerUrl` under `ws://127.0.0.1:9229/...`, so `9233` was the wrong fixed endpoint for this build; CodeStudio Lite's scan range starting at 9229 is the right verification target.
- After adding a Claude login state, the window title changed to `New chat - Claude` and the UIA tree includes authenticated chat controls such as `Mode`, `Chat`, `Cowork`, `Code`, `Collapse sidebar`, and `Search`, but the debugger path is still discoverable by UIA as `Menu` -> `Developer` -> `Enable Main Process Debugger`.
- The hardened Windows debugger script now invokes the UIA-discovered `Menu`, `Developer`, and `Enable Main Process Debugger` elements through InvokePattern/TogglePattern when available, keeping DPI-aware coordinate clicks only as a fallback. Live verification with the embedded script extracted from `src-tauri/src/core/claude_desktop_patch.rs` reopened the official Node inspector at `http://127.0.0.1:9229/json` under the logged-in UI.
- The Windows Claude menu structure is stable enough for a language-independent debugger fallback: the top-level app menu is a right-side column of `MenuItemView` items with `ExpandCollapsePattern`, where the Developer submenu is the fourth item; after expanding it, the Main Process Debugger item is the first `MenuItemView` checkbox with `TogglePattern` in that submenu column. This lets the script keep English/Chinese labels as a fast path but fall back to control structure when UIA names are localized to an unknown language.
- Enabling Windows Claude Main Process Debugger opens an extra same-process `Chrome_WidgetWin_1` window with an empty title and an approximately `802x602` client rect. Plain `PostMessage(WM_CLOSE)` does not reliably hide it, but UIA `WindowPattern.Close()` does; the Node inspector endpoint on `127.0.0.1:9229` remains available after closing the window.
- Windows Claude Desktop menu localization was missing because the existing main-process injection localized only the macOS application menu. The Electron menu logic returned early unless `process.platform === "darwin"`, so Windows' in-window/hamburger app menu still showed native English labels such as `File`, `Edit`, `View`, `Developer`, `Help`, and `Enable Main Process Debugger`.
- Windows menu localization now lives in the same non-destructive Node-inspector runtime injection as the renderer localization. The injection version was bumped to `9` and installs Windows-only hooks for `Menu.buildFromTemplate`, `Menu.setApplicationMenu`, and `Menu.prototype.popup`, then relabels menu items according to the current locale before the menu is set or shown.
- Live verification against the current Claude main-process inspector on `127.0.0.1:9229` confirmed the injected runtime reports version `9` with `popupHook`, `buildHook`, and `setHook` all installed. After switching the runtime to `zh-CN`, a real Electron `Menu.buildFromTemplate` call in Claude's main process produced top-level labels `文件`, `编辑`, `视图`, `开发者`, `帮助` and Developer submenu labels `启用主进程调试器`, `显示开发者工具`, `打开应用配置文件...`.
- Windows debugger automation must not use screen-coordinate mouse fallback. Coordinate clicks can target whatever app the user is currently operating, while UI Automation actions scoped from `AutomationElement::FromHandle($window.Hwnd)` target Claude controls directly. The current implementation no longer imports Win32 mouse APIs or `System.Windows.Forms`, no longer reads UIA element rectangles for menu actions, and uses only UIA Invoke/ExpandCollapse/Toggle/Window patterns for the Claude menu route.
- Repackaging can fail to surface new Claude localization logic even when the disk cache is correct if the already-injected main-process singleton reuses itself by `CSL_INJECTION_VERSION` only. The durable freshness check should include a content signature over the generated injection wrapper, runtime script, and locale payloads, and it should dispose old timers/listeners before reinstalling when the signature changes.
- Phase 25 root cause: the Windows debugger automation saw a `Developer` element in Claude's main window before the hamburger menu was actually opened. That element was `ControlType.Button` / `ClassName=SubmenuButton` with `ExpandCollapsePattern`, so `Open-ClaudeMenu` incorrectly returned true and the later `Invoke-Element` failed. The fix is to treat the menu as open only when a non-main same-process popup root exposes a real `MenuItemView` / `ControlType.MenuItem` Developer item, then select the Main Process Debugger item only as a popup `MenuItemView` / `ControlType.CheckBox` with `TogglePattern`.
- Live Phase 25 verification on 2026-06-24: before running the fixed embedded script, `http://127.0.0.1:9229/json` timed out. After running it, the endpoint returned the Node inspector target `electron/js2c/browser_init` with a `ws://127.0.0.1:9229/...` websocket URL. The new `windows-main-debugger.log` showed the script selected the in-window `Menu` button, then the popup `Developer` `MenuItemView`, then the popup `Enable Main Process Debugger` checkbox and toggled it on.
- Phase 26 investigation: the occasional Claude promo/ad overlay is not present in the local `app.asar` shell bundle in an identifiable way; extracting and searching the installed `Claude_1.15200.0.0_x64__pzs8sxrjxfjjc\app\resources\app.asar` mostly found local shell/dependency code, so the overlay is likely rendered by the remote `claude.ai` page. The close strategy therefore must use UI Automation structure rather than local resource strings.
- Phase 26 fix: Windows debugger automation now runs `Close-ClaudeBlockingOverlayWindows` after closing Inspector/Debugger prompt windows and before opening the in-window menu. It only acts when the main Claude window contains promo/upgrade/announcement-like text, then invokes an unnamed `ControlType.Button` with `InvokePattern` as the anonymous X close button. The Mac path is unchanged because it uses the system menu/Accessibility route rather than the in-window hamburger menu.
- Phase 27 root cause: Windows Claude local utility windows and Tray menus were outside part of the existing locale refresh surface. DevTools windows could use `chrome-devtools://` URLs and title changes after creation, while Tray context menus never passed through the Windows `Menu.buildFromTemplate` / `Menu.prototype.popup` hook once assigned through `Tray.prototype.setContextMenu`.
- Phase 27 fix: the main-process runtime injection now treats both `devtools://` and `chrome-devtools://` as DevTools, resyncs DevTools titles on language changes and window title/load/devtools-opened events, and installs a Windows Tray context-menu hook that localizes menus on `setContextMenu` plus re-translates known Tray menus when the selected locale changes. The hook keeps original labels on each item so switching away from zh-CN restores English/original labels.
- Phase 28 root cause: the awkward strings "通过繁琐的任务坚持" and "船只特征，而非线条" came from static Claude Ion zh-CN locale entries, not runtime DOM fallback. The affected keys are `4ahpF5N/t0` and `ye9sGm7rX3` in `src-tauri/resources/claude-desktop/i18n/ion-dist/i18n/zh-CN.json`.
- Phase 28 fix: `4ahpF5N/t0` is now "推进繁琐任务" and `ye9sGm7rX3` is now "发布功能，而不是堆代码行数". A Rust regression test asserts these exact strings and rejects the old literal terms "坚持", "船只", and "线条" for these entries.
- Phase 29 root cause: Windows debugger automation could spend its first seconds in AppX activation (`Get-AppxPackage` / `Get-StartApps` / `Start-Process shell:AppsFolder`) even when a Claude window was already open, so user focus changes could delay the later UIA menu actions. Inspector prompt cleanup also ran as fixed single scans before and after the flow, so a prompt that appeared shortly after the toggle could remain visible.
- Phase 29 fix: the Windows UIA script now looks for an existing Claude main window before AppX activation, only starts the AppX app if no window is found, shortens fixed waits around focus/menu/developer expansion, and adds `Wait-CloseClaudeInspectorPromptWindows` polling around the debugger toggle and confirmation loop. The no-coordinate/no-mouse safety boundary remains in place.
- Phase 30 root cause: CodeStudio Lite route-entry refreshes are intentionally asynchronous, but Codex Client's late `applyState()` replaced `settingsDraft` with the scanned backend settings even if the user changed launch/settings controls during the refresh. The Settings page had the same stale-load shape: `loadAppSettings()` could return after a user changed language/theme/auth-preservation and reset the visible controls to old values.
- Phase 30 fix: Codex Client refresh now records `settingsSaveRevision` at refresh start and applies scan results with `preserveDraft` whenever the revision changes before each late write. It still updates scan/install state, but keeps the current draft/save status/stale plan marker. Settings now uses a small edit revision guard so stale initial loads update only the backing `settings` object and do not overwrite newer UI edits.
